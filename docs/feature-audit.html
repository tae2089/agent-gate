우선 검토할 4곳

코드 줄 수만으로 판단하지 않았습니다. 중복된 권한, 측정되지 않은 효과, 사용하지 않는 호스트, 토큰 비용을 함께 봤습니다.

1

세 runtime을 실제로 모두 쓰는가?

배포하지 않는 host 지원은 보호 범위를 넓히지 않고 adapter와 테스트만 유지시킵니다. 사용 host가 확인되면 가장 큰 삭제 후보입니다.

2

watermark + reinject가 재개 오류를 줄였는가?

둘은 한 순환을 이루므로 따로 평가하면 안 됩니다. 긴 작업의 재질문·잘못된 재개율을 플랫폼 summary만 쓴 경우와 비교해야 합니다.

3

readiness judge가 설계 누락을 줄였는가?

artifact-judge와 skill-verifier는 호출과 점수를 보장하지만 재작업 감소 효과는 운영 데이터가 없습니다. 판정 번복·재작업·토큰을 함께 측정해야 합니다.

4

시나리오 추적 완성도 100%의 범위는?

모든 필수 시나리오의 전용 검사가 최신 상태에서 통과했다는 뜻입니다. 시나리오 완전성이나 전체 코드 품질 점수가 아닙니다.

Decision history

제거·미도입 결정

활성 footprint에 포함하지 않습니다. 같은 기능을 이름만 바꿔 다시 도입하지 않도록 대체 경계와 재도입 조건을 남깁니다.

  • removedprompt-router

    모델의 기본 skill 선택과 Stop verifier 사이에서 같은 정책을 한 번 더 알렸지만 재작업 감소 근거가 없어 제거했습니다.

  • removedscenario rollout modes

    advisory·critical·enforce 분기를 없애고 opt-in이면 하나의 fail-closed 정책만 사용합니다.

  • removedchild overlay · inheritance · parent-candidate

    작업마다 직접 task·flow·scenario를 두고 병렬 작업은 별도 worktree로 격리합니다.

  • removedrunner registry · review

    별도 registry와 LLM 검토를 제거하고 scenario가 shell 없는 argv를 직접 선언합니다.

  • removedJUnit XML / test-report rule

    언어별 report parser를 skill verifier에서 제거하고 실제 scenario command의 exit code와 evidence를 사용합니다.

  • not adoptedstage1-autorun

    편집마다 전체 검증을 반복하지 않습니다. 완료·CI 경계의 scenario run이 실행 증거를 소유합니다.

  • not adoptedscope-gate

    파일 범위 DSL과 child 정책을 만들지 않습니다. active design, scenario 완료, diff review로 대응합니다.

Removal protocol

기능을 지우기 전 확인할 것

코드 줄 수나 개인 취향이 아니라 실제 보호 계약과 관찰된 가치로 결정합니다.

  1. 이 기능이 막은 실제 실패 사례가 있는가?
  2. 같은 불변식을 이미 다른 gate가 소유하는가?
  3. 기능이 없어도 동일한 실패 동작을 보장할 수 있는가?
  4. 사용하지 않는 host·mode·artifact만 잘라낼 수 있는가?
  5. 삭제 후 어떤 회귀 시나리오로 안전을 증명할 것인가?
  6. 재도입 조건을 숫자나 관찰 사건으로 적을 수 있는가?
